# Mimesis Engineering Working Spec

This is the working design spec for turning Mimesis Engineering from a strong idea into a public GitHub system, launch asset, and operating method.

## 1. Intent

Mimesis Engineering is not a prompt collection.

It is a way to turn strong external artifacts into operating standards for AI-native work.

The practical goal is simple:

> Do not ask AI to act like an expert. Show it the artifacts experts actually made, extract the structure, then use that structure to build, judge, and revise your own work.

## 2. Public Definition

Mimesis Engineering is an artifact-level method for AI-native builders.

It uses real products, papers, repos, blogs, user flows, visual systems, proof surfaces, and failure cases as reference material. The method decomposes those references into patterns that can be transferred, transformed, rejected, or tested.

## 3. Private Operator Definition

For the operator, Mimesis Engineering is also a work operating system.

It connects:

- artifact analysis
- branch-based problem solving
- AI-agent role assignment
- memory capture
- proof building
- launch execution
- network absorption

## 5. Repository Roles

The prepared GitHub projects should not compete with each other. They should form a system.

| Repo | Role |
|---|---|
| `mimesis-engineering` | Source of truth, public thesis, method, launch hub |
| `mimesis-canvas` | Worksheet layer for applying the method |
| `mimesis-casebook` | Proof layer with public before/after cases |
| `harness-meta` | Operator memory, decision ledger, drift control |
| `chetdeuk-engine` | Future learning engine and habit loop |
| `market-intelligence` | Private research and channel intelligence |
| `svy04.github.io` | Public narrative, blog, proof surface |

## 6. Agent Roles

| Actor | Role | Output |
|---|---|---|
| Human operator | Direction, taste, final judgment | Intent, constraints, decisions |
| GPT | Strategy, writing, synthesis | Docs, launch copy, working specs |
| Gemini | Skeptic, comparison, critique | Failure checks, blind spots, alternatives |
| Codex | Builder | Files, repo structure, issues, automation |
| Hermes | Orchestrator | Daily plan, memory log, backlog, handoff |

## 7. Core Loop

1. Map the field.
2. Identify strong artifacts and failures.
3. Model their structure.
4. Extract transferable patterns.
5. Synthesize a variation under the operator's constraints.
6. Inspect failure modes.
7. Ship a proof artifact.
8. Record sensation, knowledge, and wisdom.
9. Merge the winning branch into the system.

## 8. Memory Categories

| Category | Meaning | Example |
|---|---|---|
| Sensation | What people reacted to | “Artifact-level imitation” landed better than “expert prompt” |
| Knowledge | What became reliable | README must show the artifact/canvas/case loop in 30 seconds |
| Wisdom | What should guide the next move | Launch only when proof artifacts are ready, not when the idea feels exciting |

## 9. Launch Thesis

The launch is not about pretending to be popular.

The launch is about compressing real discovery into one visible window.

When AI performance hype rises, the public question becomes: “How should people actually work with these models?”

Mimesis Engineering should answer:

> Stop prompting. Start cloning workflows.

## 10. Success Criteria

A successful v0.1 launch means:

1. Visitors understand the method in 30 seconds.
2. Builders can try it in 5 minutes.
3. At least two cases show concrete before/after changes.
4. GitHub traffic turns into stars, issues, discussions, forks, DMs, or contributor candidates.
5. The operator captures the launch into sensation, knowledge, and wisdom.

## 11. Non-Negotiable Guardrails

Do not build fake attention.

Do not use fake stars, fake comments, fake accounts, copied comments, or engagement bait.

Do not frame Mimesis as proven beyond the current evidence.

Do not copy protected work. Extract structure, name influences when appropriate, and maintain claim boundaries.
