# Launch Strategy: Mimesis Drop

This is the launch strategy for concentrating legitimate attention around Mimesis Engineering.

The goal is not fake popularity.

The goal is compressed discovery:

> Many real people discovering one real artifact in one visible window.

## 1. Launch Thesis

The timing window opens when public attention shifts from “AI is impressive” to “How do we actually work with this?”

When that happens, Mimesis Engineering should appear as the answer:

> Stop prompting. Start cloning workflows.

Meaning:

- stop relying on persona prompts
- stop asking models to pretend
- start using strong artifacts as standards
- start extracting structure
- start building proof surfaces
- start turning good work into repeatable operating loops

## 2. Launch Objective

The first public drop should create three outcomes.

| Outcome | Meaning |
|---|---|
| Recognition | People remember the term `Mimesis Engineering` |
| Proof | People see cases, canvases, prompts, and boundaries |
| Absorption | Traffic becomes stars, issues, discussions, DMs, forks, contributors, and relationships |

## 3. What Must Be Ready Before Launch

Do not launch only because the idea feels exciting.

Launch when these are ready:

- README explains the idea in 30 seconds
- quickstart works in 5 minutes
- canvas is linked clearly
- casebook has at least 2 usable cases
- proof boundary is visible
- launch posts are prepared
- response scripts are ready
- internal supporters have actually used or reviewed it
- there is a way to capture feedback
- Hermes has a launch-day logging routine

## 4. Ethical Attention Rules

Allowed:

- ask real users to try it
- ask supporters for honest feedback
- ask people to share only if useful
- ask people to open issues if confused
- ask people to describe their real use case
- concentrate publishing across channels

Not allowed:

- fake stars
- fake comments
- fake accounts
- copied comments
- pretending unrelated people are organic fans
- hiding conflicts of interest
- claiming proof that does not exist

The launch should feel intense because the artifact is ready and the timing is right, not because the engagement is manufactured.

## 5. Channel Architecture

GitHub is the source of truth.

Every other channel should drive people back to the repo or the proof surface.

| Channel | Job | Message Style |
|---|---|---|
| GitHub | Trust, star, fork, issue | practical, structured, proof-aware |
| X | FOMO, discovery, narrative | sharp, short, high-energy |
| LinkedIn | credibility, networking | strategic, business-readable |
| Hacker News | critique, builder credibility | humble, technical, usable |
| Product Hunt | maker discovery | product/toolkit framing |
| Disquiet | Korean maker audience | direct, maker-friendly |
| OKKY | Korean developer feedback | practical, no hype |
| Velog | long-form Korean explanation | educational, beginner-friendly |
| svy04.github.io | proof surface and archive | polished, durable |

## 6. Launch Day Wave

Use KST as the operator clock.

### Wave 0 — Final Check

Before publishing:

- README links work
- canvas link works
- casebook link works
- proof boundary visible
- release note drafted
- issue templates ready
- all posts have one clear CTA

### Wave 1 — Global Discovery

Publish first wave:

- GitHub release
- X thread
- LinkedIn post
- personal site post

CTA:

> Star the repo if useful, try the canvas, and open an issue with your workflow.

### Wave 2 — Builder Critique

Publish to builder communities only when the repo is usable:

- Hacker News Show HN
- Product Hunt if it is framed as a usable toolkit
- relevant Reddit/dev communities only after reading community rules

CTA:

> I am looking for critique on the method, proof boundary, and examples.

### Wave 3 — Korean Expansion

Publish Korean explanations:

- Disquiet
- OKKY
- Velog
- Korean X/LinkedIn

CTA:

> AI를 잘 쓰는 프롬프트가 아니라, AI와 함께 일하는 운영 방식을 만들고 있습니다. 피드백 부탁드립니다.

### Wave 4 — Response and Absorption

Do not disappear after posting.

For every meaningful response:

1. reply clearly
2. capture the feedback
3. convert confusion into README/FAQ fixes
4. convert strong users into case candidates
5. convert builders into contributors

## 7. Core Copy

### English Hook

```text
AI models got better.
But most workflows did not.

Mimesis Engineering turns strong artifacts into standards for AI-assisted creation.

Do not ask AI to pretend to be an expert.
Show it what experts actually made.
```

### Short English Hook

```text
Stop prompting.
Start cloning workflows.
```

### Korean Hook

```text
AI 성능은 좋아졌는데, 사람들은 아직도 AI를 검색창처럼 쓴다.

미메시스 엔지니어링은 AI에게 전문가 흉내를 시키는 방법이 아니다.
강한 결과물의 구조를 뽑아내서, 내 작업의 기준으로 쓰는 방법이다.
```

### Builder-Friendly Version

```text
I built a small open method for artifact-level Mimesis:
map strong references, extract their structure, synthesize your version, inspect failure modes, then ship proof.

Looking for critique from builders who use AI for real work.
```

## 8. Internal Supporter Squad

The internal group should not be used as a fake engagement machine.

Use them as a launch squad with real jobs.

| Role | Job |
|---|---|
| Tester | Try canvas and report friction |
| Explainer | Rewrite confusing parts in simpler words |
| Skeptic | Attack claims and proof boundary |
| Builder | Add examples, prompts, or issue comments |
| Amplifier | Share only their real take, in their own words |

Supporter instruction:

```text
써보고 좋으면 star 하세요.
헷갈리면 issue 남겨주세요.
공유할 거면 본인 언어로만 공유하세요.
복붙 댓글은 하지 마세요.
```

## 9. Metrics

Do not judge only by stars.

Track:

- stars
- watchers
- forks
- issues
- discussions
- PRs
- case submissions
- DMs
- calls booked
- contributor candidates
- comments that reveal confusion
- comments that reveal positioning strength

## 10. Launch Success Definition

A good first drop means:

- people understand the difference from prompt engineering
- people use the canvas
- at least a few people ask for help applying it
- at least one external person opens an issue or discussion
- the maintainer gets new network connections
- the next version becomes obvious

## 11. Post-Launch Rule

After launch, update the repo within 24 hours.

Add:

- FAQ from real confusion
- top criticism
- fixed wording
- new case candidate list
- launch reflection in Sensation / Knowledge / Wisdom format
