# Contributing

Mimesis Engineering needs contributions that help people understand, use, trust, or improve the method.

The best contribution leaves an artifact trail.

## What Contributions Are Useful

- weak artifacts that need a first loop
- before/after cases
- reference packs
- adapter contracts
- plugin shapes
- boundary improvements
- confusing sections that need clearer docs
- proof notes that make claims safer

## Submit a Weak Artifact

Open a Weak Artifact issue.

Bring:

- artifact type
- current artifact
- audience
- what feels weak
- desired transformation
- references if any
- permission to turn into a public case

## Submit a Case

Open a Case Submission issue.

If the artifact comes from an external owner, customer, client, or private context, open a Permissioned External Case issue first so permission, redaction, and publication boundaries are explicit.

Use the shape:

- starting artifact
- references studied
- extracted structure
- transformation
- before / after
- proof boundary
- permission to publish

## Submit a Reference Pack

A Reference Pack should include:

- Use When
- Source Quality
- What Good Artifacts Usually Do
- What to Inspect
- Transferable Patterns
- Do Not Copy
- Starter Prompt

Source Quality should name whether the references are official docs, maintained source repos, papers, patents, standards, public artifacts, local artifacts, or secondary routing material.

## Improve the Boundary

Boundary work is core work.

Report:

- surface copying
- wording similarity
- layout similarity
- borrowed authority
- unsupported claims
- missing attribution
- shallow imitation

## Do Not Submit Copied Material

Do not paste protected works.
Do not submit examples you do not have permission to share.
Do not turn Mimesis into surface copying.

## Do Not Submit Fake Proof

Do not submit:

- fake users
- fake stars
- fake comments
- fake testimonials
- fake endorsements
- unverifiable outcomes

No proof, no claim.

## Submit An Adapter Or Plugin Shape

Adapter and plugin contributions must state their status.

Allowed statuses:

- `contract`
- `prototype`
- `usable`
- `verified`

Do not describe a contract as a shipped integration.
Do not mark a tool `verified` without a public case note.

## Case Format

Use [templates/case-note.md](templates/case-note.md).

Every case should show:

- before
- references
- extracted structure
- after
- what improved
- what remains unproven

## Pull Request Checklist

- Does this help someone understand, use, trust, or contribute to Mimesis?
- Does it avoid overclaiming?
- Does it preserve originality boundaries?
- Does it add or improve an artifact trail?
- Does it avoid fake proof?
