# Codex Master Task — Build Mimesis Engineering

Use this task when asking Codex to implement or improve the public Mimesis Engineering repository.

## Role

You are the implementation engineer for `mimesis-engineering`.

Do not treat this as a vague philosophy repo.

Treat it as a public operating method for AI-native builders.

## Goal

Make the repository understandable, usable, believable, and launchable.

A stranger should be able to:

1. understand the method in 30 seconds
2. try it in 5 minutes
3. see at least one proof case
4. know how to contribute or open an issue

## Core Definition

Mimesis Engineering is artifact-level imitation for AI-native work.

It does not ask AI to pretend to be an expert.

It shows AI strong artifacts and asks it to extract structure, judgment loops, proof surfaces, failure handling, and transferable patterns.

## Existing System

Use these repos as a system:

- `svy04/mimesis-engineering` — source of truth and public hub
- `svy04/mimesis-canvas` — worksheet layer
- `svy04/mimesis-casebook` — proof/case layer
- `svy04/harness-meta` — operator memory and decision layer
- `svy04/chetdeuk-engine` — future learning/practice engine
- `svy04/market-intelligence` — private research layer
- `svy04/svy04.github.io` — public blog and proof surface

## Build Priorities

### Priority 1 — README

Improve README so it clearly answers:

- What is this?
- Why is it different from prompt engineering?
- How do I use it now?
- Where are the canvas and casebook?
- What local artifacts show, and what remains unverified?

### Priority 2 — Quickstart

Create a 5-minute path:

1. choose a weak artifact
2. choose 3 strong references
3. decompose their structure
4. extract patterns
5. transform the weak artifact
6. inspect failure modes
7. record what changed

### Priority 3 — Proof

Add links and templates for cases:

- starting artifact
- reference artifacts
- extracted structure
- transformation
- before/after
- claim boundary
- next proof artifact

### Priority 4 — Launch Readiness

Add or improve:

- launch strategy
- traffic absorption system
- issue templates
- contribution path
- release checklist

## Writing Rules

Use simple language.

Avoid buzzwords when a simple phrase works.

Do not overclaim.

Do not say Mimesis is an industry standard.

Do not imply the method is scientifically proven unless proof is shown.

Do not encourage copying protected work.

## Output Standard

Every file should either:

- explain the method
- help someone use it
- prove it through a case
- help the maintainer operate it
- help the launch absorb attention

If a file does none of these, remove or rewrite it.

## Done Criteria

A task is done only when:

- links work
- examples are concrete
- claims have boundaries
- the next action is obvious
- a beginner can follow without private context
