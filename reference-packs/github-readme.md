# GitHub README Reference Pack

## Use When

You want to make a GitHub repo easier to understand, trust, use, and contribute to.

## Source Quality

Prefer official project repositories, maintained source repos, release docs, contribution guides, and public issue or discussion trails.
Avoid using marketing summaries as proof of how a README actually works.

## What Good Artifacts Usually Do

- explain the project quickly
- name the problem
- show what the user gets
- provide a quickstart
- show examples or proof
- state limitations
- explain contribution paths

## What to Inspect

- opening hook
- problem statement
- quickstart
- output examples
- trust devices
- proof boundary
- contribution path

## Transferable Patterns

- old way vs new way
- one-command or first-loop start
- what this creates
- before/after case
- limitation section
- contribution CTA

## Do Not Copy

- exact wording
- brand-specific claims
- logos
- visuals
- layout too closely
- proof you do not have

## Starter Prompt

```text
Study the reference READMEs below.
Extract their structure, not their wording.

Identify how they explain the problem, show the first action, build trust, state limitations, and invite contribution.

Then transform my README under my project's constraints.
```
