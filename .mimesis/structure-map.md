# Structure Map

## Knowledge Structure

The framework needs a short public promise, a concrete loop, named outputs, and a proof boundary.
The method is easier to trust when every abstract term has a file or template behind it.

## Visual / Layout Structure

README should move from hook to first loop to framework map.
Folder READMEs should act as small entry points rather than hidden directories.

## User Flow

Visitor flow:

1. Understand "Give AI standards, not roles."
2. Bring one weak artifact.
3. Pick a reference pack.
4. Fill the `.mimesis` files.
5. Improve the artifact.
6. Inspect boundary and write a case note.

## Trust Devices

Trust comes from:

- explicit proof boundary
- before/after case notes
- adapter/plugin status labels
- no fake proof language
- local file protocol that can be inspected

## Claim Boundary

Allowed:
Mimesis v0.1 defines a Markdown-first framework, file protocol, templates, reference packs, cases, adapter contracts, and plugin shapes.

Not allowed:
Claims of benchmarked productivity, external adoption, legal originality guarantee, or shipped plugins.

## Failure Handling

If a loop copies surface, hides references, or claims outcomes without evidence, the boundary check must mark the loop as incomplete.

## Rhythm / Taste

Short, declarative, field-manual style.
Mythic enough to feel like a future work system, but grounded by files and proof boundaries.

## Transferable Patterns

- first-screen slogan
- weak artifact call to action
- seven artifact outputs
- file protocol
- casebook proof surface
- contribution path
- status labels for integrations

## Do Not Copy

- protected wording from references
- visual surface of other frameworks
- external authority without proof
- fake social proof
